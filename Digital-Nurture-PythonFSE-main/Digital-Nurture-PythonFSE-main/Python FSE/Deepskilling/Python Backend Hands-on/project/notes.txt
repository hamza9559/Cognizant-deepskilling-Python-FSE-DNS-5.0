2. MIDDLEWARE IN DJANGO
--------------------------------------------------------------------------------
Middleware is a framework of hooks into Django's request/response processing. It's a 
lightweight, low-level "plugin" system for globally altering Django's input or output.

Two built-in Django middleware classes:
  - SecurityMiddleware (`django.middleware.security.SecurityMiddleware`): 
    Enhances security by handling production-level HTTP protections like X-Content-Type-Options, 
    X-Frame-Options, and SSL redirects.
  - CsrfViewMiddleware (`django.middleware.csrf.CsrfViewMiddleware`): 
    Protects the backend from Cross-Site Request Forgery attacks by adding easy-to-verify, 
    hidden tokens to state-changing requests (like POST, PUT, DELETE).


3. WSGI VS ASGI
--------------------------------------------------------------------------------
  - WSGI (Web Server Gateway Interface): 
    The classic, synchronous Python standard for web applications. It handles requests 
    one at a time per worker thread. Django uses WSGI by default (`wsgi.py`).
  - ASGI (Asynchronous Server Gateway Interface): 
    The modern successor to WSGI. It allows asynchronous processing, enabling a single 
    worker to handle thousands of concurrent background operations without blocking.
  
  When to switch to ASGI? 
  You switch to ASGI (`asgi.py`) when your Django application requires real-time, long-lived 
  connections like WebSockets, Server-Sent Events (SSE), background tasks, or high-concurrency async streaming.


4. MVC PATTERN MAPPED TO DJANGO'S MVT
--------------------------------------------------------------------------------
The classic MVC (Model-View-Controller) design pattern maps directly to Django's 
MVT (Model-View-Template) pattern, though the naming can trick you at first glance:

  - Model (MVC) -> Model (MVT): 
    Maps directly. Defines the data layout, database abstractions, and core business rules.
  - View (MVC) -> Template (MVT): 
    The visual layer presented to the user (HTML/JSON formatting).
  - Controller (MVC) -> View (MVT): 
    This is the core shift. Django's framework handles the URL routing (the Controller role), 
    and your custom 'View' acts as the bridge that fetches data from the Model and decides 
    which Template/Response to deliver.
